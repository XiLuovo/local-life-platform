# 面试问答速记版

## 项目一句话

我做的是一个本地生活平台后端，把外卖系统和点评秒杀系统整合成了一个统一业务平台，覆盖门店浏览、优惠券秒杀、外卖下单、社交笔记和签到等场景。

## 项目亮点一句话

我重点做了 Redis 相关能力，包括缓存、BitMap 签到、ZSet 点赞和 Feed 流，以及 Lua + Redis Stream 的高并发秒杀异步下单。

## 你为什么做这个项目

因为原来的两个项目各有优势，一个业务链路完整，一个技术亮点强。我想把它们整合成一个既能展示完整业务，又能体现 Redis 和高并发能力的项目，更适合实习面试。

## 你主要负责什么

- 业务整合和模块设计
- 用户登录与 JWT 鉴权改造
- 门店、优惠券、秒杀券和券订单
- 探店笔记、关注和签到
- Redis 高并发秒杀方案

## 为什么把微信登录改成手机号验证码登录

因为微信登录依赖外部环境，本地演示成本高。改成手机号验证码登录后，更容易跑通完整链路，也更方便后续接社交和营销模块。

## 为什么不直接做微服务

因为当前目标是做一个适合实习展示的完整项目。单体结构更适合快速打通业务闭环，也能更集中地展示后端开发和 Redis 高并发能力。等主链路稳定后，再拆微服务更合理。

## 秒杀怎么防止超卖

我先用 Lua 脚本在 Redis 里原子完成库存校验和一人一单判断，校验通过后再扣减 Redis 库存并写入 Stream。后台消费者异步落库时再做一次数据库扣减和订单校验，减少超卖风险。

## 为什么用 Redis Stream

因为秒杀请求量大会冲击数据库。Redis Stream 可以把请求先缓存成消息，再由后台异步处理订单，起到削峰填谷的作用，而且还能通过 pending-list 做失败补偿。

## 点赞和 Feed 为什么用 ZSet

因为点赞和 Feed 都有明显的时间顺序需求。ZSet 可以按 score 排序，既方便存时间戳，也方便做最近点赞用户查询和滚动分页。

## 签到为什么用 BitMap

因为签到天然是“某一天签了还是没签”的二值状态，用 BitMap 存储很省空间。再配合 bitfield，就可以计算连续签到天数。

## 这个项目最难的地方

最难的不是单个功能，而是把两个不同风格的项目整合成一个统一故事，同时保证业务链路完整、技术亮点清晰、结构还能稳定运行。

## 你最大的收获

最大的收获是学会了怎么把一个练手项目做成可展示、可讲述、可扩展的后端项目，而不是只停留在“实现几个接口”。

## 如果继续优化你会做什么

- 补博客评论和更完整的社交闭环
- 完善测试和压测
- 给秒杀链路加监控和限流
- 进一步拆分营销和社交模块

## 结束时可以怎么说

这个项目让我从单纯写功能，进步到开始思考业务整合、技术选型和系统设计，所以我也希望在实习里继续把这些能力放到真实业务中打磨。
