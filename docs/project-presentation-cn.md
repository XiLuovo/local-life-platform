# 本地生活平台：项目展示与面试说明

本文用于从简历和面试视角介绍项目。所有技术结论均应能回溯到仓库代码、测试、压测报告或 CI 配置；压测数据是本机单节点 Docker 环境结果，不代表生产容量。

## 30 秒项目定位

这是一个以 Spring Boot 为核心的本地生活平台后端，覆盖外卖、商家管理、门店营销、点评社交和高并发秒杀。项目的重点不是简单拼接业务接口，而是围绕秒杀链路补齐原子准入、Redis Stream 异步落库、有限重试、`XCLAIM` 故障接管、幂等终结、保守补偿、DLQ、Testcontainers 集成测试、k6 压测和 Prometheus 可观测性，形成一条可以运行、验证和解释的工程闭环。

## 简历项目描述

以下 4 条可以直接放入中文简历，再根据版面删减：

1. 基于 Java 8、Spring Boot 2.7.3、MyBatis、MySQL 与 Redis 构建本地生活平台后端，整合外卖下单、商家管理、门店优惠券、点评社交、签到与秒杀业务，并使用 Maven 多模块管理公共能力、领域对象和服务端实现。
2. 设计 Redis Lua + Redis Stream 秒杀链路：Lua 原子完成库存校验、一人一单、资格预扣、`PENDING` 状态和消息投递；后台消费者在事务中异步扣减数据库库存并创建订单，通过状态查询 interface 向客户端暴露最终结果。
3. 加固异步消息可靠性：实现延迟有限重试、`XPENDING + XCLAIM` 崩溃接管、Lua 幂等终结和 DLQ；重试耗尽后先核对数据库最终状态，仅对确定性冲突执行补偿，对不确定结果保留资格并转人工复核，避免错误回补造成超卖。
4. 建立可复现验证闭环：使用 Testcontainers 覆盖 MySQL 8、Redis 7、成功下单、重复拒绝、人工复核、`XCLAIM` 和 Lua 终结幂等；使用 k6 完成 100/500/1000 并发本机压测，1000 并发场景达到 1098.90 req/s、P95 118.72 ms、异步清空 8.20 s，并通过 MySQL、Redis、Stream 一致性校验；通过 Micrometer/Prometheus 观测准入、处理、重试、接管、pending 和 DLQ。

### 数据口径提醒

- 1098.90 req/s 和 P95 118.72 ms 衡量的是 Redis Lua 准入与 Stream 投递，不是同步 MySQL 写入。
- 8.20 s 是 1000 并发场景在请求突发结束后，500 个已准入订单完成异步落库并确认消息所需的清空时间。
- 测试环境为 Windows 10、i7-11800H、31.6 GB 内存、Docker Desktop 单机拓扑；简历或面试中不能将结果描述为生产 QPS。
- 100、500、1000 并发三组结果分别为 666.67、998.00、1098.90 req/s，P95 分别为 128.52、267.97、118.72 ms；不要只挑数字后声称延迟随并发单调变化。

## 3 分钟项目介绍稿

### 0:00 - 0:30：项目背景

我的项目是一个本地生活平台后端，使用 Java 8 和 Spring Boot 2.7.3，业务上覆盖外卖、商家管理、门店与优惠券、点评社交、签到和秒杀。相比只完成 CRUD，我重点完善了秒杀异步链路，因为它能同时体现 Redis 原子操作、消息可靠性、数据一致性、测试、压测和可观测性。

### 0:30 - 1:20：核心链路

客户端发起秒杀后，服务先从数据库校验优惠券和活动时间，再生成订单 ID。随后执行 Redis Lua，原子完成库存检查、一人一单检查、Redis 库存预扣、用户购买资格记录、订单状态写入 `PENDING`，以及向 `stream.orders` 投递订单消息。准入成功后接口直接返回订单 ID，客户端通过状态 interface 查询 `PENDING`、`SUCCESS` 或 `FAILED`。

后台消费者从消费组 `g1` 读取消息，并在数据库事务中检查同一订单和一人一单、扣减数据库库存、写入订单。成功后由 Lua 原子写入 `SUCCESS`、`XACK` 并清理重试键。因此，HTTP 准入 interface 保持较小，而库存预扣、消息投递和状态初始化等复杂 implementation 被收敛在 Lua 中；MyBatis 是数据库 seam 上的持久化 adapter。

### 1:20 - 2:10：可靠性设计

如果数据库处理暂时失败，消息留在 pending list，消费者按固定间隔进行有限重试，避免立即反复读取形成热循环。如果某个消费者崩溃，其他实例会通过 `XPENDING` 找到超过空闲阈值的消息，再用 `XCLAIM` 接管。

重试耗尽后不会直接回补库存，而是先查询数据库最终状态。如果同一个订单已经存在，就按幂等成功处理；如果明确是用户已有该券订单或数据库库存耗尽，再执行确定性补偿；如果无法确认最终状态，则保留 Redis 预扣和购买资格，把消息幂等写入 DLQ 等待人工复核。这样牺牲了一部分库存可用性，换取不因错误回补造成超卖。

### 2:10 - 2:40：验证和观测

我用 Testcontainers 启动隔离的 MySQL 8 和 Redis 7，通过公开 HTTP interface 测试成功下单、重复拒绝、数据库异常后的人工复核，以及崩溃消费者消息被 `XCLAIM` 接管。同时测试 Lua 终结脚本第一次执行和重复执行的幂等语义。

可观测性使用 Micrometer 和 Prometheus，记录准入结果与耗时、异步处理结果与耗时、重试耗尽、`XCLAIM`、消费者异常、pending 数和 DLQ 长度。标签只使用固定枚举，不放用户 ID、订单 ID 或异常文本，避免高基数和敏感信息泄露。

### 2:40 - 3:00：结果和边界

k6 在本机单 Docker 拓扑下执行独立 50 VU 预热和 100、500、1000 并发场景。1000 并发时，500 个库存全部被正确准入，另外 500 个返回售罄，准入吞吐为 1098.90 req/s、P95 为 118.72 ms，异步清空耗时 8.20 s；MySQL、Redis 和 Stream 的一致性检查全部通过。我会明确说明这是可复现的本机实验结果，而不是生产容量承诺。

## 核心设计取舍

| 设计问题 | 当前选择 | 收益 | 代价与后续方向 |
| --- | --- | --- | --- |
| 秒杀接口是否同步写 MySQL | Redis Lua 准入后异步写库 | 缩短同步链路，把数据库写入与突发请求解耦 | 客户端需要轮询状态，系统需要处理最终一致性 |
| 准入操作如何保证原子性 | 单个 Lua 内完成校验、预扣、状态初始化和 `XADD` | 避免多条 Redis 命令之间出现部分成功 | Lua interface 必须保持稳定，脚本升级需要兼容现有 key 与消息字段 |
| 数据库失败后是否立即回补 | 先重试和核对最终状态，只补偿确定性冲突 | 避免“数据库其实已提交但响应丢失”时错误回补导致超卖 | 不确定结果会暂时占用库存，需要 DLQ 和人工复核 |
| pending 消息如何恢复 | 当前消费者延迟重试，其他实例通过 `XCLAIM` 接管陈旧消息 | 同时处理瞬时故障和消费者崩溃 | 阈值过小可能抢占仍在处理的消息，过大则延长恢复时间 |
| 消费并发模型 | 每个 JVM 一个单线程消费者，多个 JVM 可横向扩展 | 单实例内顺序清晰，故障与幂等逻辑易验证 | 单实例异步清空速度受限；更高负载下可评估受控线程池或分片消费 |
| 状态如何返回客户端 | Redis Hash 保存状态，缺失时按当前用户回源 MySQL | 快速轮询并防止越权查询，Redis 状态丢失后仍可确认成功订单 | 需要 TTL 与历史状态保留策略；失败详情必须避免泄露内部信息 |
| 指标标签如何设计 | 只使用固定 outcome/event 枚举 | 控制 Prometheus 基数，避免标识符和异常文本泄露 | 细粒度问题需要结合结构化日志或追踪系统定位 |
| 管理端点如何暴露 | 仅启用 health/prometheus，非 Docker 默认绑定 `127.0.0.1` | 减少 Actuator 暴露面 | 生产环境仍需内网、网关、鉴权或防火墙保护 |

## 故障场景与系统行为

| 场景 | 检测方式 | 当前处理 | 关键不变量 |
| --- | --- | --- | --- |
| Redis 库存为 0 | 准入 Lua 返回售罄码 | 请求失败，不写 Stream | 不产生无库存订单 |
| 用户重复抢同一券 | Lua 检查 buyer Set | 请求失败，不重复预扣 | 一人一单 |
| 数据库临时异常 | 消息处理抛异常 | pending 保留，按间隔有限重试 | 未成功前不 `XACK` |
| 消费者处理后崩溃 | 消息长时间停留在其他消费者 pending | `XPENDING + XCLAIM` 接管并重放 | 重放不能产生重复订单或重复终结副作用 |
| 同一订单已落库 | 按订单 ID 与用户查询到精确记录 | 幂等写 `SUCCESS` 并确认消息 | 已成功订单不能被改成失败 |
| 用户已有同券订单 | 按用户和券查询到其他订单 | 恢复本次多占的 Redis 库存，保留用户购买资格，写 DLQ | 不重复创建订单，不释放真实购买资格 |
| 数据库库存已耗尽 | 条件扣减影响行数为 0 | 将 Redis 库存校正为 0、释放本次资格并写 DLQ | Redis 不继续放行无数据库库存请求 |
| 重试耗尽且无法确认数据库状态 | 最终状态检查无结果或检查失败 | 不回补、不释放资格，写人工复核 DLQ | 优先避免错误回补导致超卖 |
| Stream 消息字段损坏 | 无法映射订单字段 | 幂等写入 malformed DLQ，`XACK` 并清理重试键 | 坏消息不阻塞主消费循环 |
| Redis 状态丢失 | 状态 Hash 为空 | 按订单 ID 与当前用户回源数据库 | 不向其他用户泄露订单 |
| Gauge 刷新失败 | pending/DLQ 查询抛异常 | 保留上一次值并增加 `state_refresh_error` | 指标采集失败不影响交易主链路 |

## 高频面试追问

### 1. 为什么不用数据库悲观锁直接做秒杀？

数据库条件扣减仍然是最终库存约束，但如果所有突发请求都同步竞争数据库，会拉长接口时延并把峰值压力直接传给 MySQL。当前设计先由 Redis Lua 快速完成准入，再通过 Stream 异步落库。代价是必须显式解决最终一致性、状态查询、重试和补偿，而不是声称 Redis 替代了数据库约束。

### 2. Lua 已经保证一人一单，为什么数据库还要再查？

Redis 是准入层，不是唯一事实来源。消息可能重放，Redis 数据也可能被清理或恢复，因此数据库事务仍检查精确订单和用户券关系。Redis 校验减少无效流量，数据库约束和查询保护最终结果，两者职责不同。

### 3. Stream 消息会不会丢？

Lua 将资格预扣、状态初始化和 `XADD` 放在同一次 Redis 原子执行中，避免“预扣成功但没有消息”。消费端只有终结 Lua 成功后才 `XACK`。消费者崩溃时消息留在 pending list，可由其他实例 `XCLAIM`。这不是跨 Redis 与 MySQL 的严格分布式事务，因此仍通过幂等、状态核对和保守补偿收敛最终结果。

### 4. 为什么重试耗尽后不统一恢复库存？

因为异常可能发生在数据库提交成功但应用未收到成功结果之后。如果此时恢复 Redis 库存和购买资格，其他用户可能再次抢到已经在数据库中消耗的库存，造成超卖。因此必须先查最终状态；无法确认时宁可暂时少卖并进入人工复核，也不冒险多卖。

### 5. `XCLAIM` 会不会让两个消费者同时处理？

消息可能因为超时阈值和长事务出现重复处理，所以不能把 `XCLAIM` 当作“绝不重复”的保证。系统依靠订单 ID 查询、一人一单检查、数据库事务和 Lua 终结幂等承受重放。claim idle 阈值也必须大于正常处理时间并结合监控调整。

### 6. 为什么还需要 DLQ？

有限重试只能处理瞬时故障，无法安全解决格式损坏、确定性业务冲突或数据库状态不确定。DLQ 把这些终止条件从主消费循环隔离出来，保留原消息、原因和订单信息，便于人工或补偿任务后续处理，同时防止坏消息无限阻塞。

### 7. 1000 并发为什么 P95 反而低于 500 并发？

这是本机 Docker 单次实验结果，受容器调度、JIT、连接复用和运行时噪声影响，不能据此推断并发越高延迟越低。报告保留三组原始 k6 摘要，正确表述应是“在给定环境中得到这些观测值”，若要比较趋势应重复多轮并报告分布和置信区间。

### 8. 当前设计最大的可扩展性限制是什么？

每个 JVM 使用一个单线程 Stream 消费者，逻辑简单且便于验证，但单实例的数据库落库吞吐有限。从 1000 并发场景 8.20 s 的异步清空时间也能看到同步准入与异步处理能力并不相同。后续可以评估多消费者、受控线程池或按券分片，但必须先保证数据库连接池、锁竞争和幂等逻辑能承受更高并发。

### 9. Prometheus 为什么不使用 orderId 或 userId 标签？

这些值的数量会持续增长，放入标签会造成高基数时间序列并增加存储和查询成本，同时可能暴露业务标识。当前指标只按固定 outcome/event 枚举聚合；具体订单问题应通过受控日志、审计记录或追踪系统定位。
