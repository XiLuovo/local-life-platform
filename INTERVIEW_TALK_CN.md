# 面试讲解稿（中文）

## 项目定位

这个项目我做的是一个本地生活平台后端。最开始仓库里其实有两个独立项目，一个偏外卖业务，一个偏点评和秒杀。我没有简单把两个作业拼在一起，而是重新按业务闭环做了整合，让它变成一个可以讲“门店浏览、领券秒杀、下单支付、社交互动”的完整平台。

## 1 分钟版本

我做的项目是一个 Spring Boot 的本地生活平台后端，核心是把外卖系统和点评秒杀系统整合成一个统一业务。用户可以手机号登录后浏览门店、查看优惠券、参与秒杀、下外卖订单，还可以发探店笔记、点赞、关注和签到。技术上我重点做了 Redis 相关能力，包括门店缓存、BitMap 连续签到、ZSet 点赞和 Feed 流，以及 Lua + Redis Stream 的高并发秒杀异步下单，这部分是我认为最有代表性的技术亮点。

## 3 分钟版本

这个项目原本来源于两个独立练手项目，一个是苍穹外卖，业务主线比较完整，包含员工、菜品、套餐、购物车、订单、报表这些典型外卖场景；另一个是黑马点评，亮点更多在 Redis、高并发和社交玩法，比如优惠券秒杀、探店笔记、关注流和签到。

我在整合时主要做了三件事。

第一，统一业务主线。我保留了外卖系统原有的多模块结构，把它作为主骨架，然后把点评项目里的门店、优惠券、秒杀、笔记、关注和签到能力接入进来，形成一个完整的本地生活平台，而不是两个分散的 demo。

第二，统一用户体系。原来的用户端主要依赖微信登录，这对本地演示和面试不太友好，所以我把它改成了手机号验证码登录 + JWT 鉴权。这样项目既能跑通，又方便我继续接社交、签到和营销模块。

第三，补技术亮点。我把秒杀券下单从同步实现升级成了 Redis Lua + Stream 异步方案。Lua 脚本先在 Redis 层原子完成库存校验和一人一单判断，再写入 Stream；后台消费者异步处理订单落库，并处理 pending-list 补偿。这套方案能比较清楚地体现我对高并发和异步削峰的理解。

如果面试官想继续深挖，我通常会重点展开秒杀链路、Redis 在社交模块里的使用方式，以及为什么我选择“先做单体业务整合，再谈微服务拆分”。

## 我在项目里重点负责什么

- 业务整合与模块拆分
- 用户登录与鉴权改造
- 门店、优惠券、秒杀和券订单模块

- README、接口和演示链路整理

## 最好讲的技术亮点

### 1. 秒杀下单为什么用 Lua + Redis Stream

因为秒杀请求量大，如果直接查库扣库存，很容易出现超卖和数据库压力过大。Lua 脚本能把“校验库存、判断一人一单、扣减 Redis 库存、写消息到 Stream”放到 Redis 里原子执行，避免并发竞争。后续订单落库由消费者异步处理，实现削峰。

### 2. 社交功能里为什么用 Redis

- 点赞：用 ZSet 存用户点赞时间，既能判断是否点过赞，也能查最近点赞用户。
- 关注：用 Set 存关注关系，方便求共同关注。
- Feed：关注作者发笔记后，把笔记 id 推送到粉丝的 ZSet 中，用户查收件箱时做滚动分页。
- 签到：用 BitMap 记录每天是否签到，再通过 bitfield 统计连续签到天数。

### 3. 为什么把微信登录改成手机号验证码登录

主要是为了降低项目运行门槛，让项目在本地环境就能完成完整业务演示。对于实习面试来说，这样比依赖微信小程序生态更实用，也更容易讲清楚用户体系和鉴权设计。

## 面试官可能会问的问题

### Q1：你为什么不直接做微服务？

我的考虑是，这个阶段目标是做一个面向实习、可完整演示的项目。单体结构更利于快速打通业务闭环，也方便我集中展示 Redis、高并发和业务设计能力。等主链路稳定后，再拆分服务会更自然。

### Q2：你怎么防止秒杀超卖？

先在 Redis Lua 脚本里原子判断库存是否充足，再判断用户是否重复下单，校验通过后才扣减库存并写入消息队列。数据库层面异步落库时也会再次做库存扣减和重复单校验，形成双层保护。

### Q3：为什么点赞和 Feed 选 ZSet？

因为点赞和 Feed 都天然带时间顺序，ZSet 支持 score 排序，既适合按时间维护顺序，也方便分页和查询最近记录。

### Q4：这个项目最难的地方是什么？

最难的是把两个不同风格的项目整合成一个统一故事，同时保证结构可运行、业务链路完整、技术亮点还足够突出。不是简单迁代码，而是要重新决定哪些能力保留、哪些能力重构、哪些能力先做一个适合展示的版本。

## 你可以这样收尾

这个项目对我来说最大的收获，不只是写了几个功能，而是学会了怎么把一个“能跑的练手项目”整理成“有完整业务闭环、有技术亮点、能在面试里讲清楚的项目”。
